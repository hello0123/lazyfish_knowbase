## 階段 0（第 1 週）：骨架、DB schema、Docker、pytest、CI

**目標**：把 `POST /articles` 和 `GET /articles/{id}` 兩個端點做出來，資料先進 DB 就好，中間的爬蟲跟 AI 摘要都先跳過。

### 詳細步驟

1. **建立專案設定檔**
   - `pyproject.toml`：定義套件名稱、runtime 依賴（fastapi、uvicorn、sqlalchemy、pydantic）、dev 依賴（pytest、httpx、ruff、black），以及 ruff/black 的 lint 規則（line-length 100、目標 Python 3.11）。
   - `.gitignore`：排除 `__pycache__/`、`.venv/`、`*.db`、`.pytest_cache/`、`.ruff_cache/` 等不該進版控的檔案。

2. **依「職責」切分程式碼（Single Responsibility Principle）**
   - `app/db.py`：`engine`、`SessionLocal`、`Base`、`get_db()` 這個 generator（給 FastAPI 依賴注入用）。
   - `app/models.py`：SQLAlchemy ORM models——`Article`、`Tag`，以及兩者的多對多 junction table `article_tags`。
   - `app/schemas.py`：Pydantic schema——`ArticleCreate`（含 `source_type` 的 `field_validator`）、`ArticleResponse`（`from_attributes=True` 讓它能直接讀 ORM 物件）。
   - `app/services/llm_provider.py`：先建一個抽象類別 `LLMProvider(ABC)`，定義 `summarize` / `extract_tags` 兩個介面，實作留到之後幾週。
   - `app/routers/articles.py`：兩個端點的商業邏輯——`POST /articles` 做欄位驗證、寫入 `status="pending"` 的紀錄、回傳 201；`GET /articles/{id}` 查不到就回 404。
   - `app/main.py`：組裝入口，建表（`Base.metadata.create_all`）、掛上 router、加一個 `/health` 健康檢查端點。

3. **寫測試**（`tests/`）
   - `conftest.py`：用 `tmp_path` 建一個獨立的 SQLite 檔案當測試資料庫，透過 `app.dependency_overrides[get_db]` 把正式的 DB 換掉，確保測試不會動到真正的 `knowledge_base.db`。
   - `test_articles.py`：6 個測試，涵蓋新增文章（url / note 兩種來源、缺欄位、`source_type` 打錯字）跟查詢文章（查得到 / 查不到）。

4. **建立本地開發環境並驗證**
   ```powershell
   python -m venv venv
   .\venv\Scripts\Activate.ps1
   pip install -e ".[dev]"
   ruff check .
   pytest -v
   ```
   結果：ruff 無警告、6 個測試全數通過。

5. **補齊部署與 CI 骨架**
   - `Dockerfile`：`python:3.11-slim` 基底，安裝套件、複製程式碼、`uvicorn` 啟動。
   - `docker-compose.yml`：掛載 `./data` volume，設定 `DATABASE_URL` 環境變數。
   - `.github/workflows/ci.yml`：push 到 `main` 或 `feat/**`、或對 `main` 開 PR 時，自動跑 `ruff check .` + `pytest -v`。
   - `README.md`：簡單的安裝、開發、測試指令說明。

6. **手動驗證整條路徑（透過 `/docs` Swagger 介面）**
   - 啟動伺服器：`uvicorn app.main:app --reload`
   - `POST /articles`（`source_type: "url"`）→ 收到 `201`，`status` 為 `"pending"`，`summary` 為 `null`（因為摘要功能還沒接）。
   - `GET /articles/{id}` → 拿到跟剛剛 POST 回傳完全一致的資料，確認資料真的落地到 SQLite 檔案，不是只存在記憶體。
   - 確認 422 驗證機制正常運作。

7. **Git／GitHub**
   - 原規劃是「開 `feat/project-skeleton` branch → 完成骨架 → 開 PR 給自己合併」，但實際操作時直接在 `main` 上 `git init` → `git add` → `git commit`（commit `a61ad6c`）→ `git remote add origin` → `git push`，等於**跳過了 branch + PR 這一步**。
   - 討論後的結論：第一個骨架 commit 直接上 `main` 是常見的 bootstrap 做法，不用回頭補；但從下一個功能（也就是階段 1）開始，正式建立 `feat/*` branch → PR → merge 的習慣。

### 階段 0 完成時的已知限制

- 丟網址進去不會真的抓取內容，只是把網址字串存起來。
- `summary`、標籤都是空的，因為 `llm_provider.py` 還只是抽象介面，沒有實作。
- 沒有非同步背景處理，`status` 永遠停在 `"pending"`。
- Docker 的部分寫好了但還沒有實際 `docker-compose build/up` 驗證過。
